# Спека: спавн листьев (`sub_415AB0` / `sub_4163B0`) и глобальный список листьев `dword_4D866C`

Цель: зафиксировать “истину” о том, **как и когда** в оригинале создаются листовые секции (`TreeSectionLeaf`), какие поля они инициализируют, и как работает глобальная структура “лист-очереди” (`dword_4D866C`), которая затем обслуживается в кадре.

---

## 1) Точка входа: bud → twig (`sub_415C10`) спавнит листья

Файл: `decompiled/sub_415C10.c`

В конце успешной конверсии bud → twig, после привязки нового twiг’а к bud и переноса части полей:
- `sub_415AB0(a1, a2);`

Где:
- `a1` — бывший Bud-объект (после конверсии часть его состояния сбрасывается),
- `a2` — параметр, вычисленный в `sub_415EF0` как нормализованная энергия bud:
  - `a2 = clamp01((budEnergy - minBudEnergy) / (maxBudEnergy - minBudEnergy))`

Важно: есть дополнительное условие “срезать” родителя при включённом глобальном тумблере:
- если `*(dword_4D7EE8 + 220) != 0` и `bud+516 > 2` и родитель валиден по `byte_4D8225[...]`,
  - вызывается `(*parent + 44)(parent)` и конверсия прекращается.

## 2) `sub_415AB0(a1, a2)` — создаёт 2 листа вокруг bud

Файл: `decompiled/sub_415AB0.c`

Поля bud, используемые/меняемые:
- `a1+512` — угол (в радианах), вокруг которого раскладываются листья:
  - `a1+512 += (rand*(0.9..1.1))*π/2`
  - нормализация по \(2π\)
- `a1+516` — счётчик листьев: `++*(a1+516)`

Алгоритм:
- вычисляет ориентационный вектор `v6` через `sub_4158D0(v6, 0, 1, angle)` (по сути “ось/направление” от угла)
- создаёт `TreeSectionLeaf` через `operator new(0x220)` и конструктор `sub_4163B0(parentTwig, newLeaf, a2, v6)`
- затем повторяет второй раз с `angle + π` (лист напротив)

## 3) `sub_4163B0(thisTwig, leafObj, a3, a4)` — конструктор `TreeSectionLeaf`

Файл: `decompiled/sub_4163B0.c`

Ключевое:
- ставит vftable `TreeSectionLeaf::vftable`
- выставляет `leafObj+4 = 7` (type id)
- заполняет цветовые множители `leafObj+512..524 = 1`
- `leafObj+452`:
  - `leafObj+452 = lerp(flt_4D85F0, flt_4D85F4, a3)`,
  - затем умножается на рандом \(0.9..1.1\)
- `leafObj+428 = 0`
- добавляет запись в **глобальный список листьев** `dword_4D866C`:
  - делает `node = sub_40FD70(dword_4D866C, tail, &payload)`
  - вставляет `node` в двусвязный список
  - `++dword_4D8670`

Payload, который кладётся в node — это pointer-поле (`a4` в декомпиле), фактически объект/идентификатор, связанный с leaf (в `sub_4163B0` это отражено криво из-за артефактов декомпиляции; важно, что список хранит value в `node+8`).

## 4) `dword_4D866C` / `dword_4D8670` — структура и жизненный цикл

Инициализация:
- `sub_470360()` выделяет “sentinel” node (`operator new(0xC)`) и делает цикл:
  - `dword_4D866C->next = dword_4D866C`
  - `dword_4D866C->prev = dword_4D866C`
  - регистрирует `atexit(sub_472120)`

Очистка:
- `sub_472120()` вычищает все nodes и удаляет sentinel, обнуляя `dword_4D8670`.

Node формат (по `sub_40FD70`):
- 12 байт: `{ next, prev, value }`

## 5) Обслуживание очереди листьев: `sub_40E0A0`

Файл: `decompiled/sub_40E0A0.c`

Поведение:
- если `dword_4D8670 != 0`:
  - берёт `value = (tail->value)` у последнего элемента,
  - удаляет последний элемент списка и декрементит `dword_4D8670`,
  - если `value != 0` и `value+180 != 0` (есть parent),
    - вызывает `sub_40E230((float*)value, dword_4D7EE8)`
- иначе (`dword_4D8670 == 0`):
  - делает проход по списку `dword_4D8C10` и для объектов подходящих по `byte_4D8225[...]` добавляет в лист-очередь значения `v2[46]`.

Интерпретация:
- `dword_4D866C` — **очередь/лист задач**, из которой в кадре вытаскивают один элемент и обрабатывают `sub_40E230`.
- `sub_4163B0` при создании leaf кладёт элемент в эту очередь.

OPEN:
- точная семантика `sub_40E230` (что делает с leaf/родителем/миром) — надо разобрать, это ключ к “lifetime” листа после появления.

## 6) `sub_40E230(leaf, world)` — потребитель очереди листьев (lifetime шаг)

Файл: `decompiled/sub_40E230.c`

Ключевое поведение:
- читает INI `directLightPercent` в `flt_4D8CF0` (lazy init)
- генерирует случайную ориентацию (yaw/pitch) и строит матрицу `D3DXMatrixRotationYawPitchRoll`
- вычисляет случайный вектор направления `v20` (через `sub_401540`), а с вероятностью `directLightPercent` подменяет его на “прямой свет” (`sub_408590(&flt_4D538C, ...)`)
- берёт позицию leaf из `a1[2..4]` и смещает её по некоторому направлению, зависящему от `a1[5]`:
  - `sub_40D610(...)` + `sub_401500(..., ..., a1[5])` и прибавление к позиции
- вызывает `sub_40E460(&pos, world, &dir, 1.0)`

Вывод:
- очередь листьев `dword_4D866C` обслуживается так, что каждый тик берётся один leaf-like объект,
  и для него запускается трассировка/освещение/рост-обработка через `sub_40E460` (с направлением, иногда совпадающим с глобальным светом).

OPEN:
- `sub_40E460`: это фактический “leaf lifetime/lighting” процессор (его нужно разбирать отдельно).

## 7) `sub_40E460(pos, world, dir, intensity)` — трассировка/затухание света вдоль направления

Файл: `decompiled/sub_40E460.c`

Сигнатура (по декомпилю): `sub_40E460(float *pos, int world, float *dir, float intensity)`

Высокоуровневая роль:
- делает “ray-march/пробу” по дереву вдоль `dir` начиная из `pos`, и по пути **модифицирует** `intensity` (затухание/экранирование).
- по окончании (и на каждом шаге при дебаге) может рисовать отрезки, если включён `byte_4D7E8D`.

Ключевые шаги:
1) Строит сегмент/луч:
   - `p0 = pos + dir * 200`
   - `p1 = pos - dir * 200`
   (через `sub_401470(..., dir, 200.0)` + `sub_401430/sub_4013F0`).

2) Собирает пересечения/кандидаты с деревом:
   - `n = sub_450E30(world->root, outHits, p0, p1, 30)`
   - каждый hit — тройка, где `hit[0]` это `sectionPtr` (смещение `*(_DWORD*)v8`), а `hit[1]` используется как дистанция/параметр шага.

3) Для каждого hit:
   - двигает “сопротивление/вектор” секции (поля `section+196..204`) к направлению `dir`:
     - скорость зависит от `(section+208 + 0.01) * 10`
     - `section+196..204 = normalize( sectionVec * (rate) + dir*(intensity+0.01) )` (по сути через `sub_401430`/`sub_40CF00`).
   - считает `dot = sectionVec · dir`, берёт `x = clamp01(-dot)`.
   - обновляет 2 сглаженных “лайт-аккумулятора” секции:
     - `section+208 = (x*intensity + section+208*10) / 11`
     - `section+212 = (intensity + section+212*20) / 21`

4) Дальше — затухание `intensity` по типу секции:
   - если `byte_4D8227[11*type(section)] != 0` (один класс объектов):
     - берёт `abs( dot( sub_40D6D0(section), dir ) )` и применяет затухание:
       - INI `lightDecayAmount` → `flt_4D62EC` (default в `.data`: 0.4)
       - `intensity = (1 - flt_4D62EC * absDot) * intensity`
   - иначе, если `byte_4D822A[...] != 0` (другой класс объектов):
     - делает геометрический тест “луч пересёк/прошёл достаточно близко”:
       - строит `v64` из `sub_40D6C0`, берёт `radius2 = (section+444)^2`
       - если `radius2 > sub_44EE20(lineSegment(p0,p1), v64)` → экранирование:
         - `intensity = intensity * 0.5`

5) Debug draw:
   - если `byte_4D7E8D` — рисует последовательные отрезки `sub_44E5D0` цветом `(intensity,intensity,intensity, 0.3)`.

Вывод:
- `sub_40E460` — это **проход по секциям вдоль направления**, который:
  - обновляет “light-response” поля секции (`+208/+212`, и вектор `+196..204`),
  - и уменьшает входной `intensity` от шага к шагу (затухание/экранирование).

OPEN (чтобы закрыть на 100%):
- семантика `byte_4D8227` (какие типы секций подпадают под затухание через `lightDecayAmount`).

### 7.2) Что такое `byte_4D8227` (наблюдаемая “истина”)

`byte_4D8227` — это **не отдельный массив**, а срез внутри 11-байтовой таблицы флагов типов:
- базовый адрес: `byte_4D8220`
- шаг по типу: `11 * typeId` (везде в коде)
- `byte_4D8227 == byte_4D8220 + 7` (7-й байт в строке типа)

То есть проверки вида:
- `byte_4D8227[11 * *(obj+4)]`
означают “у данного `typeId` включён флаг #7”.

В `sub_40E460` флаг #7 выбирает ветку затухания через INI `lightDecayAmount`.
В `sub_40F140` флаг #7 влияет на характер обновления world-object (больше рандом-дребезга и меньше стабильного падения).

OPEN:
- где именно инициализируется флаг #7 для конкретных `typeId` (скорее всего через общий механизм конфигов/регистрации типов).

Текущий вывод по дампам (важно):
- в доступных декомпилированных функциях и в `.text.txt` не найдено ни одной инструкции, которая **пишет** в `byte_4D8227[...]` или вообще в диапазон `0x004D8220..0x004D822A`, кроме `byte_4D8220[11*type]=1` (через `sub_401730`) и `byte_4D8225[11*type]=1` (через `sub_403A90`).
- следовательно, в рамках имеющихся данных мы не можем перечислить `typeId` с флагом #7: либо таблица заполняется косвенно (например через memcpy/инициализацию данных без явных символов), либо флаг #7 по умолчанию остаётся 0.

Рантайм-проверка (IDA Debugger, write-watch):
- поставлен брейкпоинт на write в диапазон `0x004D8220..0x004D822A` (len=11),
- срабатывание происходит на `.text:00401793`:
  - `imul eax, 0Bh`
  - `mov byte_4D8220[eax], bl`
  - стек: `sub_401830 -> sub_401730 -> start -> wWinMain`
- в момент срабатывания дамп `0x004D8210..` показывает нули (таблица ещё не содержит выставленных флагов типа `+7`).

Промежуточный вывод:
- единственная подтверждённая ранняя запись в таблицу — это `byte_4D8220[11*type]=1` (flag #0).
- **записи в `byte_4D8227` (flag #7) не наблюдались** даже на старте процесса.

Практическое следствие для TS-переноса (до появления новых доказательств):
- ветка `byte_4D8227` в `sub_40E460` следует считать **практически недостижимой**,
  и моделировать поведение света/затухания через ветку `byte_4D822A` (геометрическое экранирование) как основную.

Уточнение по рантайм-watch на `0x004D8227` (важно):
- в коде адрес считается как `0x004D8227 + 11 * typeId` (см. `imul eax, 0Bh` перед доступом).
- **write-breakpoint на один байт `0x004D8227` ловит только `typeId = 0`**, а не лист (`TreeSectionLeaf` имеет `this+4 == 7`).
- для листа нужно смотреть **`0x004D8227 + 11*7 = 0x004D8274`**, либо ставить watch на диапазон `0x004D8220..` (например `0x80` байт) в режиме **Write**.

Рантайм-подтверждение (закрывает leaf-часть `byte_4D8227`):
- при write-breakpoint на `0x004D8274` (это `byte_4D8227` для `typeId=7`) срабатывание происходит в инициализации `sub_40FAD0`:
  - `.text:0040FB42 mov byte_4D82A3, bl`
  - `.text:0040FB48 mov byte_4D8274, bl`
- `sub_40FAD0` вызывается lazily при `dword_4D8CD8 & 1 == 0` и используется в leaf-пайплайне (см. `sub_4151B0`, `sub_40FC70`, `sub_40F140`, `sub_40E460`).

Вывод:
- для `TreeSectionLeaf` (`typeId=7`) флаг #7 (`byte_4D8227[11*7]`) **действительно устанавливается в 1**.
- следовательно, в `sub_40E460` ветка “затухание через `lightDecayAmount`” **достижима** как минимум для `typeId=7`.

### 7.1) Формат `hits` из `sub_450E30` (что мы знаем точно)

`sub_450E30(a1, outHits, p0, p1, maxHits)`:
- подготавливает глобальные:
  - `dword_4D82B4 = maxHits`
  - `dword_4D82B0 = outHits`
  - `dword_4D82B8 = 0` (count)
- вызывает рекурсивный сбор `sub_450CD0(section, p0, p1, (p0-p1), len2)`
- затем сортирует массив `outHits` по компаратору `sub_450E10`:
  - сравнение: `hits[j].field1 > hits[i].field1` (сортировка по убыванию второго float)

Каждый hit занимает **12 байт** (3 * 4 байта):
- `hit[0]`: `sectionPtr` (int)
- `hit[1]`: float, по коду записи `sub_450CD0` это `sqrt(len2) * sectionRadius` где `sectionRadius = *(section+20)`.
  - В `sub_40E460` это используется как **шаг вдоль `dir`** (`sub_401470(dir, hit[1])`) перед пересчётом позиции/состояния.
- `hit[2]`: float, `sqrt(dist2)` — расстояние от сегмента луча до точки `section+24` (точка/центр), если оно меньше `*(section+36)`

Примечание:
- `sub_450860` вычисляет `dist2` от точки к отрезку (классическая “distance point to segment” в квадрате).
- `sub_450970` возвращает такой же `dist2` и дополнительно пишет параметр `t` (проекция), но в `sub_450CD0` параметр `t` в hit не сохраняется.
