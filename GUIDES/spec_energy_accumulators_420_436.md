# Спека: энергетические аккумуляторы `+420` (production) и `+436` (spent)

Цель: зафиксировать “истину” о том, какие сабы в оригинале **пишут** поля секции:
- `this+420` — вклад в *производство/приток* энергии (в `sub_40DC90` прибавляется к `v0+196`)
- `this+436` — вклад в *расход/списание* энергии (в `sub_40DC90` вычитается из `v0+196`)

Контекст:
- `sub_40DC90` после вызова virtual `(*root + 36)` делает:
  - `E = E - (root+436) + (root+420)`
  - затем `E *= energyLeak`

---

## 1) Общие правила агрегации: `sub_414E10(this)`

Файл: `decompiled/sub_414E10.c`

Перед обходом детей:
- `this+420 = 0`
- `this+436 = 0`
- также сброс ряда других сумматоров (`+424`, `+480`, `+484`, `+436` и т.п.)

На каждом ребёнке:
- ребёнку выставляется доля `child+432` от `parent+432`
- вызывается виртуальный `(*child + 36)` (update по типу)
- затем **суммируются**:
  - `this+420 += child+420`
  - `this+436 += child+436`
  - `this+480 += child+480`, `this+484 += child+484`
  - `this+436 += child+436` (см. выше)

Вывод:
- `+420/+436` — это **деревовидные аккумуляторы**: листовые/конечные узлы формируют свои значения, а родители их суммируют.

## 2) Кто пишет `+436` (расход)

### 2.1 `sub_418BD0(this)` — расход на рост длины твика

Файл: `decompiled/sub_418BD0.c`

Ключевое:
- `this+436 = min(this+432, flt_4D8630)` (т.е. clamp по maxBudEnergy)

Далее:
- по `(this+436 - minBudEnergy)/(maxBudEnergy-minBudEnergy)` считается прогресс длины,
- и длина `this+448` увеличивается.

Интерпретация:
- Пока `TreeSectionTwig` растёт по длине (`this+512 != 0` и `sub_417C90` вызывает `sub_418BD0`), он выставляет `+436` как **“стоимость/используемая энергия”** на elongation, ограниченную `maxBudEnergy`.

### 2.2 `sub_417620(this)` — расход на спавн побега из листа

Файл: `decompiled/sub_417620.c`

При успешной попытке создать новый побег:
- `this+436 += flt_4D862C` (minBudEnergy)

### 2.3 `sub_417FF0(a1)` — расход на апикальное/билатеральное ветвление

Файл: `decompiled/sub_417FF0.c`

Если доступный headroom достаточен:
- вычисляется `v17 = apicalGrowthStrength * ((a1+432 - a1+436) * apicalFactor)`
- если `v17 >= minBudEnergy`:
  - `a1+436 += v17`
  - дальше создаются новые `Bud`/ветви (через `sub_4159C0`, `sub_4158D0`) с распределением энергии.

### 2.4 `sub_4188E0(a1)` — расход на боковую ветку (lateral spawn)

Файл: `decompiled/sub_4188E0.c`

Если `headroom = (a1+432 - a1+436) >= minBudEnergy` и проходит вероятность:
- выбирается `v23` в диапазоне `[minBudEnergy..min(headroom,maxBudEnergy)]`
- `a1+436 += v23`
- создаётся `Bud` (`sub_4159C0`) и выставляется начальная энергия/радиус новой ветви.

## 3) Кто пишет `+420` (производство/приток)

Факты из декомпила (закрыто):
- `sub_413F50` и `sub_414E10` сбрасывают `+420` в 0 и далее **суммируют детей**.
- `sub_40DC90` добавляет `root+420` обратно в глобальную энергию `v0+196`.
- `sub_417C90` читает `this+420` как вход для роста радиуса (`v2 = this+420`).

Главный writer production найден: **`sub_416510` (leaf update)**.

Ключевой фрагмент (`decompiled/sub_416510.c:276–287`):
- `v16 = this+111 * v34`
- `v18 = this+114`
- `v19 = v16 * v34 * v18 * flt_4D8634`
- `*((float*)this + 105) = v19`  → это **`this + 420`**
- `*((float*)this + 106) += v19` → это **`this + 424`**

Интерпретация:
- `this+420` — “производство за тик” (мгновенный вклад) от листа/секции, масштабированный:
  - `growthRate` (`this+111`)
  - `v34` (световой фактор 0..1)
  - `energy/health` (`this+114`)
  - `leafEnergyProduction` (`flt_4D8634`)
- `this+424` — накопительный интеграл production (накапливает `+420`).
