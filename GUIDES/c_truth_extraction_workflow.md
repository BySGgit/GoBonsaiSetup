# Извлечение «истины» из декомпила C перед переносом на TypeScript

Цель: **сначала** восстановить понятную спецификацию оригинала (данные, потоки, инварианты), **потом** переносить в `gobonsai-web`. Так мы не тащим в TS артефакты Hex-Rays и не путаем «похожий код» с «тем же поведением».

---

## 1. Иерархия источников (чему верить)

| Приоритет | Источник | Зачем |
|-----------|----------|--------|
| 1 | **Поведение оригинального .exe** (если есть доступ) | Эталон; всё остальное проверяется через него |
| 2 | **`full.c`** (или единый дамп с кросс-ссылками) | Глобалы, `flt_*`, `dword_*`, порядок инициализации, дубли вызовов |
| 3 | **`decompiled/sub_XXXXXX.c`** | Локальная логика саба, но с шумом |
| 4 | **Псевдокод + ручная реконструкция** | После снятия артефактов — то, что реально переносим в TS |

Декомпил **не** является «истиной» сам по себе: он — зашумлённая проекция бинарника.

---

## 2. Типовые артефакты декомпилятора

Имеет смысл вести для каждого саба мини-таблицу «артефакт → как снять».

| Симптом | Что часто происходит | Как искать правду |
|---------|----------------------|-------------------|
| Локальные **не инициализированы** (`v49`, `v50`) | Перекрытие стека, неверное имя, или вывод в другой слот | Сверка с **`full.c`** вокруг тех же строк; поиск **того же смещения ESP** у других имён; при необходимости **листинг asm** вызова |
| **`LODWORD(float)`** в вызове | Передаётся **float**, не `int` | Считать аргументом **float** (угол, `t` для slerp и т.д.) |
| **`sub_XXX()` без аргументов** при `__usercall` / регистрах | Аргументы в **ecx/edx/esi** | Смотреть сигнатуру в начале `.c` и **предыдущие строки** (кто кладёт в регистры) |
| **`double` / `__int64` у векторов** | Склейка двух `float` или баг типов | Перечитать как **два float** или **Vector2/3** |
| **«Мёртвый» вызов** (результат не используется) | Ошибка имени буфера или оптимизация | Проверить, не **тот же буфер** под другим именем; иногда код реально мёртв после оптимизации |
| **Виртуальный вызов** `(*(void**)(*this+44))(this)` | Таблица методов | Найти класс по **смещению vtable** в `full.c` / xref |
| **`this + N` в полях** | Смещение в **байтах** | `N * 4` для `DWORD`, осторожно с `float` и выравниванием |

## 3. Процесс на одну функцию `sub_XXXXXX`

Рекомендуемый порядок (можно копировать как чеклист в issue/PR или отдельный `.md` на саб).

1. **Идентификация**
   - Имя файла, адрес, **кто вызывает** (grep по `full.c` / `sub_*.c`).
   - Соглашение о вызове: `__thiscall`, `__cdecl`, `__usercall`, какие регистры.

2. **Входы и выходы**
   - Явные параметры + **this**.
   - Побочные эффекты: какие **глобалы**, какие поля **this + offset**.

3. **Константы**
   - Все литералы и `flt_*` — выписать и сопоставить с **`full.c`** / `GrowthConstants` (одно имя — одно число).

4. **Вызовы внутрь**
   - Для каждого `sub_YYYYYY`: одна строка «зачем» (вектор, acos, slerp, …). При сомнении — открыть **`sub_YYYYYY.c`**.

5. **Псевдокод без шума**
   - Переписать блоками: *«если энергия … то обновить …»*, без `goto LABEL_49`, с **осмысленными именами** (`worldUp`, `bendAxis`, `slerpT`).

6. **Неразрешённые места**
   - Явно пометить: `OPEN: источник v49/v50` / `OPEN: порядок умножения кватернионов` — чтобы при переносе в TS не «угадывать молча».

7. **Критерий готовности к TS**
   - Есть псевдокод, таблица полей **this**, список констант, список **OPEN** с планом проверки (тест, сравнение с exe).

## 4. Процесс на подсистему (дерево / рост / метаболизм)

- Собрать **граф вызовов** от точки входа кадра (или от `sub_416510` и т.д.) **на уровне имён сабов**.
- Для каждого узла графа — либо ссылка на готовый лист разбора (п. 3), либо статус «не разобрано».
- Зафиксировать **инварианты**, которые должны сохраняться между кадрами (энергия [0,1], порядок обхода секций, что пишется в **prevData** до/после slerp).

Так мы не переносим «кусок `sub_416510`» изолированно от того, **кто задаёт матрицу +352** и **когда обновляется мир**.

## 5. Выходные артефакты (что считается «готово»)

Минимум перед переносом в TS:

| Артефакт | Содержание |
|----------|------------|
| **Spec одной функции** | Псевдокод + входы/выходы + `this` offsets + глобалы |
| **Таблица констант** | Имя в C → значение → поле в TS config |
| **Список OPEN** | Нерешённые места и как их закрыть (asm, тест, запись с exe) |
| **Связь с TS-модулем** | Напр. «этот spec → `MetabolismService` + `Sub416510Rotation`» |

Перенос в TS делается **по spec**, а не по копированию строк из `.c`.

## 6. Приоритет для GoBonsai (предложение)

Имеет смысл разбирать **снизу вверх** по зависимостям:

1. Математические примитивы: `sub_401540`, `sub_401120`/`sub_401180`, `sub_4010C0`, `sub_408470`, `sub_40FDE0`.
2. Крупный кадровый саб дерева/секции: **`sub_416510`** (метаболизм + цвет + вращение + рост) — разбить на логические блоки с отдельными spec.
3. Геометрия / ветвление: сабы из migration guide (`sub_4093B0`, `sub_4188E0`, …) по мере появления в графе вызовов.
