# Спека: `sub_40EEE0(this)` — отложенная обработка “помеченной” секции (`this+236`)

Цель: зафиксировать “истину” о том, что происходит после вызова `(*this + 44)(this)` для `TreeSectionLeaf/Twig/Stump/base`, который ставит флаг `this+236 = 1`.

Ключевой факт:
- `.text:00414260` (`vtable +44` для base/leaf/twig/stump) делает `this+236 = 1`.
- Этот флаг “пожинается” не сразу, а в специальных обходах: `sub_414270` и `sub_418E40` вызывают `sub_40EEE0(this)` **после** прохода по детям, если флаг выставлен.

## 1) Где вызывается `sub_40EEE0`

### 1.1 `sub_414270(this)` (base vtable слот; см. `full.c:16638+`)

Алгоритм:
- Обойти детей `this+168` и для каждого вызвать виртуальный `(*child + 48)` (по vtable это слот `+48`).
- После обхода: `if (*(_BYTE *)(this + 236)) sub_40EEE0(this);`

### 1.2 `sub_418E40(this)` (twig vtable/проход)

Алгоритм:
- Если `this+236` выставлен, выполняет дополнительную проверку/подготовку (`sub_415650`/`sub_418D50` в некоторых случаях).
- Обходит детей и вызывает `(*child + 48)` аналогично `sub_414270`.
- Затем: `if (this+236) sub_40EEE0(this);`

### 1.3 Прямые вызовы из логики Twig

В `sub_417C90` есть прямой вызов `sub_40EEE0(this)` в ветке “случайное отмирание/сброс” при определённых условиях (см. `decompiled/sub_417C90.c:77–86`).

## 2) Тело `sub_40EEE0` (наблюдаемое поведение)

Файл: `decompiled/sub_40EEE0.c` (и `full.c:13314+`).

Функция имеет как минимум **две смысловые ветки**:

### 2.1 Ветка A: “жёсткий режим” удаления (detach + виртуальный `+8`)

Условие:
- `if (flt_4D526C == 150.0)`

Действия:
- если есть родитель `this+180`: `sub_450C80(this, parent)` (detach из списка родителя)
- затем виртуальный слот `(*this + 8)` — в контексте проекта трактуем как **деструктор/удаление** объекта

Источник `flt_4D526C`:
- это глобальный режим/переключатель, который в `sub_406770` (обработчик ввода/команд) может устанавливаться в `150.0` (case 31).
- `sub_406AC0()` возвращает `flt_4D526C == 150.0`.

Интерпретация:
- режим `150.0` — специальный “режим мгновенного удаления/обрезки” (возможно debug/инструмент/режим очистки).

### 2.2 Ветка B: “перевести в отдельный объект” (отложенная обработка, если объект ещё активен)

Условие:
- `else if (!*(_BYTE *)(this + 188))`

Семантика флага `this+188` (закрыто по писателям/читателям):
- `sub_401830` (конструктор `WorldObject`) ставит `this+188 = 0` при создании объекта.
- `sub_401A50` и `sub_40D8C0` ставят `this+188 = 1` (после инициализации/очистки списка `this+204`).
- `sub_414E10` выполняет энерго-агрегацию/обход детей **только если `this+188 == 0`**.
- `sub_40EEE0` выполняет “detach → вставка в world-list” **только если `this+188 == 0`**.
- `sub_450F00` обрабатывает глобальный список объектов и, если `obj+188 == 1`, сначала отцепляет объект от родителя, затем вызывает виртуальный `(*obj + 4)(obj, 1)`.

Вывод (рабочее имя):
- `this+188` — флаг **“detached / world-managed”**:
  - `0`: обычный объект дерева/секции, участвует в росте/энергии и может быть “детачнут” через `sub_40EEE0`
  - `1`: объект уже в режиме “мирового объекта” (обслуживается отдельно; снимается с иерархии и обрабатывается world-системами)

Наблюдаемое:
- строятся случайные векторы (через `rand()` → диапазон `[-1..1]`, затем фильтры `sub_401500/sub_401470`)
- вычисляется/копируется матрица (копируется `this+240..` в новый объект)
- объект отсоединяется от родителя, если он есть (`sub_450C80`)
- затем создаётся и вставляется узел в список, хранящийся в глобальном объекте `dword_4D7EE8` по смещениям:
  - `v12 = *(_DWORD *)(dword_4D7EE8 + 204)` — голова списка
  - `*(dword_4D7EE8 + 208)++` — размер списка
  - вставка выполняется через `sub_40FD70(...)` и линковку `*(_DWORD *)v14[1] = v14`

Сопоставление:
- `sub_40D8C0(this)` очищает список `this+204` и обнуляет `this+208`, затем ставит `*(_BYTE *)(this + 188) = 1`.
- Это подтверждает, что `+188` — флаг состояния/активности, а `+204/+208` — контейнер (list) у “мирового/игрового” объекта, которым, судя по множественным чтениям, является `dword_4D7EE8`.

Интерпретация (пока рабочая):
- `sub_40EEE0` для leaf/twig/base — это **pipeline “отсоединить секцию и передать её в отдельный список объектов”** (дебрис/отпавшие части/мировые объекты), либо “пометить на удаление” в зависимости от режима.

Уточнение (закрыто по `sub_40CD30/sub_40CC70`):
- “отдельный список объектов” `dword_4D7EE8+204` действительно **обслуживается как отдельная сцена/дерево объектов**:
  - `sub_40CD30` вызывает `sub_40CC70(root)` и затем `sub_40CC70(obj)` для каждого элемента world-list.
  - `sub_412700` вызывает `sub_40CD30` непосредственно перед `sub_4093B0` (перестройка вершин дерева), т.е. detached-объекты попадают в обработку/геометрию на том же кадре/интервале.
