# План полного технического паритета TS с оригинальным C/EXE

## 1) Цель

Достичь воспроизводимого `1:1` поведения симуляции дерева относительно оригинального `GoBonsai.exe`:

- тот же порядок вызовов в кадре (`sub_412700` -> `sub_4130D0` -> `sub_40DC90` цепочка),
- те же числовые результаты по ключевым полям (`+420/+424/+428/+432/+436/+444/+448` и глобалы мира),
- те же условия ветвления/спавна/детача,
- контролируемая визуальная разница (рендер-стек другой, но математика и состояние совпадают).

## 2) Текущее состояние (факт-база)

- В `decompiled` обнаружено `1221` функций `sub_*.c`.
- В текущей карте `GUIDES/sub_to_ts_coverage_map.md` отражено `63` `sub_*`.
- Не покрыто картой: `1158` функций (это главный источник "поздних сюрпризов").
- По текущей карте: `8` зеленых, `40` желтых, `5` оранжевых (рендер-стек отличен по дизайну).
- Критические открытые точки в TS:
  - `sub_416510` rotation: матрица `this+352` (не только quaternion),
  - `sub_4188E0/sub_417FF0`: матрица `+240`, roll `v20`, полный RNG-порядок,
  - `sub_40E0A0/sub_40E460`: очередь листьев и fallback через twig tips,
  - `sub_414CE0`: единое поле C `+428` против split-модели в TS.

## 3) Стратегия (что делать и чего не делать)

Выбираем путь: `портировать по восстановленной "истине"`, а не пытаться сначала восстановить "красивый полный C-код".

Почему:

- полный рекавери исходников из декомпила добавляет много ложной работы и не гарантирует поведенческий паритет;
- у нас уже есть рабочий pipeline `C-truth docs -> TS -> trace diff`;
- быстрее и надежнее закрывать parity по подсистемам с числовыми гейтами.

## 4) Фазы выполнения

## Фаза A. Freeze и контроль детерминизма

Цель: исключить шум и "плавающие" факторы до любых новых портов.

Шаги:

1. Зафиксировать parity-режим как основной для верификации (`strict exe path`, фикс seed, фикс dt).
2. Вынести все gameplay-стабилизаторы в явные флаги (например, cap глобального energy pool) и отключать их в parity-run.
3. Убрать тяжелые дебаг-логи из hot path по умолчанию (оставить только флагом).

Критерий выхода:

- два запуска с одинаковым seed дают бит-в-бит одинаковый trace в TS.

## Фаза B. Полная инвентаризация декомпила

Цель: закрыть "слепую зону" 1158 функций.

Шаги:

1. Сгенерировать `master manifest` всех `sub_*` из `decompiled`.
2. Разбить функции на классы:
   - runtime growth critical,
   - physics/light critical,
   - render-only,
   - UI/tools/infra.
3. Связать каждую функцию с:
   - местом вызова (xrefs/callgraph),
   - наличием/отсутствием TS-аналога,
   - статусом: `unmapped`, `mapped-approx`, `mapped-verified`.

Критерий выхода:

- 100% `sub_*` имеют класс и статус в едином manifest.

## Фаза C. Закрытие критичного parity-backlog (приоритет P0)

Цель: убрать источники основных регрессий роста/энергии.

P0 порядок:

1. `sub_416510` (rotation block, `this+352`).
2. `sub_4188E0` + `sub_417FF0` (`+240`, roll, RNG sequence).
3. `sub_40E0A0` + `sub_40E230` + `sub_40E460` (leaf queue and tracing semantics).
4. `sub_414CE0` (объединение модели `+428`).
5. `sub_4143E0` без approximate phototropism path.

Критерий выхода:

- для каждого пункта есть:
  - truth-note в `GUIDES/ida_extracted_truth_log.md`,
  - TS-патч с ссылкой на C-строки,
  - добавленный regression-тест/trace-check.

## Фаза D. Trace-driven parity gates

Цель: сделать паритет измеряемым, а не визуально-оценочным.

Шаги:

1. Расширить fixture-набор beyond day 0/31/45/46:
   - минимум 12 контрольных точек по сезонам и возрастам.
2. Для каждой точки сравнивать:
   - root: `+420/+432/+436`,
   - дерево: распределение по типам, counts, rollups,
   - мир: `simulationDay`, `energyPool`, сезонный фактор.
3. Ввести допуски:
   - `strict` для агрегатов,
   - `bounded epsilon` для float-интерполяций.

Критерий выхода:

- `GOBONSAI_EXE_PARITY=1` проходит по всем контрольным точкам.

## Фаза E. Полный технический аудит (вся декомпиляция)

Цель: завершить "полную проверку всего декомпила", как вы запросили.

Шаги:

1. Последовательно закрывать `unmapped` по классам:
   - сначала runtime-critical,
   - затем physics/light,
   - затем UI/service,
   - в конце render-only.
2. Для render-only функций фиксировать статус `intentionally divergent` (если различие стеков осознанное).
3. Любой `APPROX/TODO(original)` либо закрывается, либо переводится в явно обоснованное divergence-note.

Критерий выхода:

- нет функций со статусом `unmapped` в runtime-critical классах;
- нет `TODO(original)` в P0/P1 модулях;
- полный manifest покрывает 1221 `sub_*`.

## 5) Контроль качества и правила изменений

Обязательные правила на каждую правку:

1. Одна правка = одна цель паритета (без смешивания багфикса и рефактора).
2. В коде: ссылка на исходный саб и участок (`sub_xxxxxx.c:line-range`).
3. В тестах: минимум один числовой assert по измененному поведению.
4. В docs: обновление `sub_to_ts_coverage_map.md` и `ida_extracted_truth_log.md`.
5. Нельзя оставлять "временный стабилизатор" без флага и примечания.

## 6) Конкретный short-term backlog (следующие шаги)

1. Создать `master manifest` по всем `sub_*` и сохранить в `GUIDES`.
2. Закрыть `sub_416510` matrix `+352` до полного parity-path.
3. Закрыть `sub_4188E0/sub_417FF0` (`+240`, roll, RNG order).
4. Убрать leaf fallback через twig tips в strict-mode (`sub_40E460` path).
5. Слить `+428` модель в TS в одно поле (поведение C).
6. Добавить расширенный набор exe-fixtures (минимум 12 точек).
7. Включить nightly parity-run (`GOBONSAI_EXE_PARITY=1`) как обязательный gate.

## 7) Ожидаемый эффект

- меньше "непонятных" регрессий после каждой правки;
- сокращение времени на отладку за счет trace-gates;
- управляемый прогресс до полного технического совпадения, а не бесконечный цикл фиксов по симптомам.
