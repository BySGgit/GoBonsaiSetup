# Начальный рост от ростка (оригинал): семя → почка → тонкий побег и листья

Документ фиксирует **восстановленную по декомпилу** цепочку классов и сабов, отвечающих за «ровный тонкий росток» и дальнейшее **пропорциональное** наращивание (радиус/длина сегмента задаются полями секции и константами из INI / `sub_40D0B0`).

---

## 1. Иерархия `TreeSection*` (vtable, `decompiled/rdata.txt`)

Типы секций дерева (первый слот vftable — часто «тип» / scalar destructor hook; далее виртуальные методы):

| Класс | `this+4` (тип в коде) | Назначение (по сабам) |
|--------|------------------------|------------------------|
| `TreeSection` | 5 | Базовая секция (ствол/ветвь) |
| `TreeSectionBud` | 6 | Почка (сферическая геометрия, `sub_416030`) |
| `TreeSectionLeaf` | 7 | Лист (`sub_416510` и др.) |
| **`TreeSectionSeed`** | **8** | **Стартовый росток / семя** |
| `TreeSectionStump` | 9 | Пень |
| **`TreeSectionTwig`** | **10** | **Тонкий сегмент побега** (`sub_418F10` — цилиндр/градиент вдоль Z) |

Имена классов и смещения vtable совпадают с RTTI в `.rdata` (`TreeSectionSeed`, `TreeSectionTwig`, …).

## 2. Создание дерева и семени

### `sub_40D960.c`

- Удаляет/обходит предыдущую корневую секцию при перезапуске.
- **`operator new(0x200)`** → **`sub_417220`** — конструктор **`TreeSectionSeed`**.
- Состояние «игры» (указатель `a1`):
  - `*(a1+200)=0`, флаги в `a1+220`, **`*(float*)(a1+216)=0`**.
  - **`*(float*)(a1+196) = rand/32767 * 40 + 40`** — стартовая энергия **40…80** (в UI часто масштабируется, см. `swprintf` с `*0.048`).
- В конце **`sub_408880(v12)`** — инициализация **глобальных параметров ветра/качания** (массив `a1[k]` — не поля `TreeSection`, а **float-блок состояния сцены** по указателю `a1+236`).

### `sub_417220.c` — конструктор `TreeSectionSeed`

1. **`sub_413F50(a1, a2, &unk_4D8BA8, &flt_4D5348)`** — базовый `TreeSection`: матрица, кватернионы, нулевая «толщина» в диапазоне 420–448, кроме переопределений ниже.
2. Vtable → **`TreeSectionSeed`**.
3. **`*(a1+4) = 8`**, **`*(float*)(a1+428) = 1.0`** (радиус/масштаб толщины для логики роста).
4. **`*(a1+416) = 0`** — уровень/счётчик сегмента с нуля.
5. Компоненты «ориентации» в **196…212** выставлены в **0,1,1,0,1** (базис для начального направления; точная семантика — вектор/кватернион в контексте `sub_413F50`).

**Вывод:** визуально «росток» — это **корневая секция типа Seed**, а не отдельный меш вне дерева.

## 3. Первый дочерний узел: почка на семени

### `sub_417440.c` (виртуальный метод `TreeSectionSeed`, см. vtable `004B8074` / `004B8078`)

- Если **нет детей** (`this+172 == 0`) и **`*(float*)(this+432) > 0`** (толщина ствола после метаболизма / `sub_414E10`):
  - Случайный малый угол **`(±0.08, 0, ±0.08)`** в `v5`.
  - **`sub_4159C0`** — создаётся **`TreeSectionBud`**, вешается на секцию.
- Всегда в конце **`sub_414E10(this)`** — обновление толщины и связанной логики.

### `sub_4159C0.c` — конструктор почки

- **`sub_413F50`** → тип **`TreeSectionBud`**, **`*(a3+4)=6`**.
- **`++*(a3+416)`** — уровень относительно родителя растёт.
- **`*(float*)(a3+452)`** — случайный размер почки из **`flt_4D85F8` (`maxBudSize`)** и множителя **0.8** (см. `sub_406AE0` в `full.c` для INI).
- **`*(float*)(a3+444) = 0.01`** — **малый радиус** (сфера почки).
- **`*(float*)(a3+428) = 1.0`**.

**Вывод:** пока толщина **432** у семени нулевая, почка **не** создаётся; после роста толщины появляется **маленькая сферическая почка** с лёгким случайным наклоном.

## 4. Переход почка → тонкий побег (`TreeSectionTwig`)

### `sub_415C10.c` — ключевой саб «созревания» почки

`sub_415C10(a1, a2)` преобразует “почку -> тонкий побег” и включает процесс наращивания длины `twig+448`.

1) Ранний выход (контроль частоты “событий”)

- `v2 = *(a1 + 180)`: если нет нужной структуры/связи — создаётся путь через `LABEL_6` (дальше `v4=0`).
- `v4` выставляется только если `byte_4D8225[11 * *(v2 + 4)]` истинно (иначе `v4=0`).
- Если выполнено:
  - `*(_BYTE *)(dword_4D7EE8 + 220)` (глобальный флаг),
  - `*(int *)(a1 + 516) > 2` (счётчик),
  - и `v4 != 0`,
  то вызывается виртуальный метод `(*(_DWORD *)v4 + 44)` и `sub_415C10` возвращает `0` (twig не создаётся).

2) Создание `TreeSectionTwig`

- Выделяется `operator new(0x220)`.
- Вызывается **`sub_417BB0(a1+240, v6, newTwig, a2, v19)`**:
  - `v19[0] = rand/32767 * 0.1 - 0.05` (yaw), `v19[1]=0`, `v19[2]=0`.
  - внутри `sub_413F50` это используется как аргументы `D3DXQuaternionRotationYawPitchRoll` для кватернионов `twig+(304)` и `twig+(320)` (случайная ориентация вокруг одной оси).
- После конструктора: `*(float*)(twig + 444) = flt_4D861C` (начальный радиус/“girth” сегмента).

3) Связь и “включение роста”

- Twig добавляется в список `*(twig + 168)` через `sub_40FD70(..., &a1)`.
- В коде есть `D3DXQuaternionMultiply(...)`, но затем `twig+(304..332)` перезаписывается identity-набором:
  - `dword_4D53A4/dword_4D53A8/dword_4D53AC = 0`
  - `dword_4D53B0 = 1.0`
- Копируются скаляры родителя в twig: `420..424`, `416`, `428` (в т.ч. `*(float*)(twig+428)=1.0`).
- Включается флаг: `*(BYTE*)(twig + 512) = 1`.
- Ставится масштаб: `*(float*)(twig + 296) = 1.0` (и аналогично для родителя).

4) Действие после замены: `sub_415AB0(a1, a2)`

- `sub_415AB0`:
  - выбирает случайный угол, создаёт два симметричных объекта (`angle` и `angle+pi`) через `sub_4163B0`,
  - и инкрементит `*(int *)(a1 + 516)++`.

### `sub_415EF0.c` — когда вызывается `sub_415C10` и что такое `a2`

`sub_415EF0(this)` сначала вызывает `sub_414E10(this)` и далее:

- если `*(float *)(this + 108) == 0`, то вместо роста/конверсии вызывается виртуальный метод `(*this + 44)`.
- иначе вычисляется `v7 = clamp01((*(float *)(this + 108) - flt_4D862C) / (flt_4D8630 - flt_4D862C))`.
- затем проверяется наличие дочернего “следующего” узла: `v4 = sub_416300(*(this + 45))` и условие конверсии `sub_415C10(this, v7)` срабатывает только если `v4 == 0 || !*(byte *)(v4 + 512)`.

Интерпретация в терминах уже восстановленных полей TS:
- `this + 108` это `baseGrowth` (управляющая энергия/потенциал для конверсии),
- `flt_4D862C = minBudEnergy = 0.2`,
- `flt_4D8630 = maxBudEnergy = 2.0`.

### `sub_417BB0.c` — конструктор `TreeSectionTwig`

- **`*(a3+4) = 10`**.
- **`*(float*)(a3+452)`** — линейная интерполяция между **`flt_4D85E8` / `flt_4D85EC` (`minSegmentSize` / `maxSegmentSize`)** по параметру **`a4`** с **случайным** множителем **0.9…1.1**.
- Дефолтный preset из `full.c` (`sub_4701D0` -> `sub_40D0B0`):
  - `flt_4D85E8 = 2.0` (minSegmentSize)
  - `flt_4D85EC = 5.0` (maxSegmentSize)
- **`*(float*)(a3+428) = 1.0`**, флаг **`192`**.

### `sub_418F10.c` — отрисовка/геометрия побега
- Масштаб: **`444`** (радиус), **`448*0.51`** — **текущая длина** по Z (тонкий удлинённый цилиндр).
- Матрица: перенос **`(0,0,1)`**, масштаб, затем умножение на локальную матрицу секции **`this+104`**.
- Текстура/градиент через константы **`flt_4D6430`** и т.д., вывод **`sub_44F2A0`** (меш).

**Вывод:** «ровный тонкий росток» в геометрии — это **`TreeSectionTwig`**:
- радиус задаётся сразу: `twig+444 = flt_4D861C` (после `sub_415C10`);
- длина `twig+448` — накопительная величина и растёт со временем (см. `sub_418BD0`).

### Значение `flt_4D861C`

- Заполняется **только** в **`sub_40D0B0`** как **14-й параметр** (`a14`); в **`full.c`** глобал **`flt_4D861C`** **не** привязан к **`sub_406AE0`** (в блоке INI ~5360–5450 есть `minSegmentSize`, `maxBudSize`, `lengthGrowRate`, `radiusGrowRate`, … — **отдельного ключа для `4D861C` нет**).
- Дефолт: **`sub_4701D0`** → **`sub_40D0B0(..., a14 = 0.079999998, ...)`** — стартовый радиус сегмента после **`sub_415C10`**; в том же вызове **a12** тоже `0.08` (дубль по значению).
- В TS: **`GrowthConstants.FLT_4D861C_INITIAL_TWIG_GIRTH`**.

### 4.1. Почему `twig+448` растёт (а не задаётся конструктором)

- В `sub_413F50` длина `*(float*)(this + 448)` при создании секции сбрасывается в `0.0`.
- В `sub_417BB0` для `TreeSectionTwig` также не задаётся `+448` — только `+444` (радиус) и `+452` (макс. размер сегмента по росту).
- Ключевой цикл:
  - `sub_417C90(this)` (для `TreeSectionTwig` это update-ветка vtable) первым делом вызывает `sub_414E10(this)` (обновление толщины/распределений по детям),
  - далее, если `*(BYTE *)(this + 512)` истинно, вызывает `sub_418BD0(this)`.
  - `sub_415C10` специально выставляет `*(BYTE*)(twig + 512) = 1`, поэтому для нового twig в ближайших update шагах срабатывает рост `+448`.
- В `sub_418BD0`:
  - вычисляется прогресс роста через `v6 = (min(*(this+432), flt_4D8630) - flt_4D862C) / (flt_4D8630 - flt_4D862C)`,
  - увеличивается длина: `*(this + 448) += pow(clamp(v6,0..1), flt_4D8640) * flt_4D8614`,
  - если длина превысила максимум `*(this + 452)`, то длина ограничивается и `*(BYTE *)(this + 512)` сбрасывается в `0`.

Дефолтные значения (из `sub_4701D0` → `sub_40D0B0`):
- `flt_4D862C` (minBudEnergy) = `0.2`
- `flt_4D8630` (maxBudEnergy) = `2.0`
- `flt_4D8614` (lengthGrowRate) = `0.079999998`
- `flt_4D8640` (lengthGrowRatePow) = `2.0`

## 4.2. Откуда берётся `this+432` (вход в прогресс длины)

- `sub_413F50` (общий конструктор базовой секции) **инициализирует** `*(float *)(this + 432) = 0.0`.
- `sub_40DC90` (внутренний глобальный цикл “updateGrowth”) обновляет энергию и **записывает** её в `+432` для корневой секции дерева:
  - `*(float *)(*(_DWORD *)(v0 + 240) + 432) = v9;`
  - где `v9 = *(float *)(v0 + 196) * flt_4D62E0 * sub_40EE60(v0)` (по смыслу — “энергия к потреблению” с масштабированием).
- `sub_414E10` (распределение по детям) не создаёт энергию, а **раскладывает** её долями:
  - `*(float *)(child + 432) = v11 * *(float *)(this + 432)`
  - после чего дочерние twig/bud используют уже “распределённый” `+432`.
- `sub_417440` (появление первой почки) использует этот же вход:
  - условие: `if (!*(_DWORD *)(this + 172) && *(float *)(this + 432) > 0.0) { ... create bud ... }`

OPEN для TS:
- В текущем `gobonsai-web/src/engine/GrowthController.ts` “энергия для наращивания длины” частично соответствует накоплению `section.growthRate`:
  - обновление `section.growthRate` при `section.growthRate < section.maxGrowth` происходит в `updateSectionGrowth()`;
  - рост `growthRate` зависит от `section.energy` и `section.lastLightFactor` (аналог световой/энергетической головной комнаты).
- В `gobonsai-web/src/engine/TreeSection.ts` эта же величина используется для “длины/высоты” геометрии:
  - `growthProgress = this.growthRate / this.maxGrowth`;
  - `currentHeightScale` затем масштабирует сегмент по оси Y (и тем самым имитирует `twig+448` рост).

OPEN (контроль паритета):
- `sub_418BD0` в C использует не линейную нормализацию, а прогресс вида `pow(clamp((E-minBud)/(maxBud-minBud)), lengthGrowRatePow) * lengthGrowRate` и ограничивает длину сверху `*(this+452)`.
- В TS сейчас применяется линейная шкалировка `growthRate / maxGrowth`, поэтому для полной идентичности нужно заменить/дополнить вычисление длины на ту же формулу (а также корректно сопоставить `this+452` с полем TS, которое задаёт максимум длины сегмента).
- В текущей TS-логике “максимум длины” сегмента фактически соответствует `this.maxGrowth`, т.к.:
  - `growthProgress = growthRate / maxGrowth` и им определяется `currentHeightScale`;
  - достижение `growthRate >= maxGrowth` влияет на переход к следующему сегменту/ветке в `GrowthController.updateSectionGrowth()`.

## 5. Листья и «пропорциональный» рост

- **`TreeSectionLeaf`**: vtable указывает на **`sub_416380`**, метаболизм/цвет/кватернион — **`sub_416510`** (см. `rdata` слот для Leaf).
- Листья появляются как **дочерние секции** в общем списке **`this+168`**; их размер и скорость роста завязаны на **`448`**, **`444`**, энергию и INI (**`leafGrowRate`**, **`minLeafLight`** — строки в `.rdata` рядом с `sub_416510`).
- Детальный разбор листа — отдельный spec (`sub_416510`, `sub_417070` для отрисовки листа).

## 8. Связь с текущим `gobonsai-web`

- Убедиться, что **корень** создаётся как аналог **`TreeSectionSeed`** (уровень 0, толщина до появления почки).
- Параметры **тонкого сегмента**: дефолт радиуса после почки — **`FLT_4D861C_INITIAL_TWIG_GIRTH`**; при расширении конфига опираться на **`minSegmentSize` / `maxSegmentSize`** (INI) и пресет **`sub_40D0B0`**.
- Почка → побег: логика **`sub_415C10` + `sub_417BB0`** должна дать скачок **`444`** с **0.01** до **~0.08** (дефолт) и смену типа секции **6 → 10**.
