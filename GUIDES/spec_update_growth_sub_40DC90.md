# Спека: updateGrowth (`sub_40DC90`) и сезонный коэффициент (`sub_40EE60`)

Цель: зафиксировать “истину” о том, как оригинал **в кадре** вычисляет и распространяет энергию/бюджет роста (ключевое поле `root+432`), включая сезонность, утечки и ежегодные события.

## 1) `sub_40EE60(a1)` — сезонный коэффициент (month curve)

Файл: `decompiled/sub_40EE60.c`

Вход:
- `a1+216` — счётчик дней (dayOfYearAccum), обновляется в `sub_40DC90` как `+1` за вызов.

Логика:
- `year = (a1+216) / 365.0`
- `yearFrac = year - floor(year)`
- `month = yearFrac * 12.0`
- `monthIndex = floor(month)` (0..11)
- `t = month - monthIndex` (0..1)
- линейная интерполяция по таблице `flt_4B786C[12]`:
  - `return flt_4B786C[monthIndex] + (flt_4B786C[(monthIndex+1)%12] - flt_4B786C[monthIndex]) * t`

Вывод:
- сезонный коэффициент **плавно меняется** внутри года и замыкается по модулю 12.

Таблица сезонности найдена в `full.c`:
- `flt_4B786C[12] = { 0.0, 0.0, 0.5, 1.0, 1.0, 0.5, 0.1, 0.1, 0.050000001, 0.0, 0.0, 0.0 }`
- 13-й элемент в декомпиле (`flt_4B786C[13]`) — артефакт, так как в `sub_40EE60` индексация делается по модулю 12.

## 2) `sub_40DC90()` — updateGrowth (кадровый цикл энергии)

Файл: `decompiled/sub_40DC90.c`

Глобальные флаги/настройки (INI):
- `byte_4D62DC` — `updateGrowth`
- `byte_4D8CED` — `randomAutoCuts`
- `flt_4D62E0` — `energyUseRate`
- `flt_4D62E4` — `energyLeak`

Состояние “игры”:
- `v0 = dword_4D7EE8` — глобальная структура состояния (энергия, день года, root pointer и т.п.)
- `*(v0+240)` — указатель на корневую секцию дерева (`TreeSectionSeed` / `TreeSection`-иерархия)
- `*(v0+196)` — накопленная энергия (в UI масштабируется, но здесь используется напрямую)
- `*(v0+216)` — “день” (растёт на 1 за updateGrowth вызов)

### 2.1 Lazy init INI-ключей (однократно)

- `updateGrowth` -> `byte_4D62DC`
- `randomAutoCuts` -> `byte_4D8CED`
- `energyUseRate` -> `flt_4D62E0`
- `energyLeak` -> `flt_4D62E4`

### 2.2 randomAutoCuts (опционально)

Если `randomAutoCuts` включён и `rand() < 0.01`, то:
- обход списка `dword_4D8C10`,
- по условиям `byte_4D8225[...]` и наличию родителя `v2[45]`,
- вызывается виртуальный метод `(*v2 + 44)` (обрезка/удаление) с вероятностью `0.01`.

### 2.3 Инкремент дня и “годовой тик”

- `*(v0+216) += 1.0`
- если пересекли границу года (целочисленный год изменился):
  - `sub_408A30(*(v0+236))` (ежегодные цели/параметры)
  - энергия: `E = E * 0.9; E = E * (1 - flt_4D7EF4)`
  - вызов виртуального `(*root + 40)`
  - `root+428 = 1.0`

Семантика `(*root + 40)` (закрыто по `decompiled/rdata.txt`):
- `root` в `v0+240` — это `TreeSectionSeed` (корень дерева).
- для `TreeSectionSeed` слот vtable `+40` указывает на `sub_414CE0`.
- `sub_414CE0(this)`:
  - обновляет `this+428` (через `byte_4D8229/byte_4D822A` режимы и `this+424`), затем сбрасывает `this+424 = 0`,
  - и делает обход детей, вызывая виртуальный `(*child + 40)` рекурсивно.

Рабочее имя слота `+40`:
- “годовой/пост-агрегационный тик” (finalize накопителей типа `+424/+428` и рекурсивный проход по дереву).

### 2.4 Запись “энергетического бюджета” в `root+432`

- `season = sub_40EE60(v0)`
- `budget = *(v0+196) * flt_4D62E0 * season`
- `*(root + 432) = budget`

### 2.5 Вызов виртуального слота `(*root + 36)`

Сразу после записи `root+432`:
- `(*(void (__thiscall **)(_DWORD))(**(_DWORD **)(v0 + 240) + 36))(*(_DWORD *)(v0 + 240));`

Смысл: **ключевой полиморфный update дерева**. Это виртуальный слот vtable по смещению **36 байт** (9-й указатель после vtable), который **разный для разных типов секций**.

Однозначное сопоставление по `decompiled/rdata.txt`:

- **`TreeSection` (base)**: `+36` → `sub_414E10` (распределение/агрегация по детям).
- **`TreeSectionSeed`**: `+36` → `sub_417440` (создание первой `Bud` при `this+432 > 0`, затем `sub_414E10`).
- **`TreeSectionBud`**: `+36` → `sub_415EF0` (созревание/переход bud → twig через `sub_415C10`).
- **`TreeSectionTwig`**: `+36` → `sub_417C90` (update стебля; внутри вызывает `sub_414E10`, рост радиуса и при флаге — `sub_418BD0` для длины `+448`).
- **`TreeSectionLeaf`**: `+36` → `sub_416510` (метаболизм/цвет/кватернионы/часть роста; внутри вызывает `sub_414E10()`).

Вывод: `sub_40DC90` после записи `root+432` запускает **виртуальный update** дерева, а затем берёт итоговые `root+436` (расход) и `root+420` (производство) для баланса `v0+196`.

### 2.6 Возврат энергии обратно в `(v0+196)` и утечка

После `(*root + 36)`:
- `E = E - (root+436)` (потраченная энергия)
- `E = E + (root+420)` (полученная энергия)
- затем `E = E * flt_4D62E4` (energyLeak)
